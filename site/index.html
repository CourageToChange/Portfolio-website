Flagship

Self-Hosted Home Server & Security Lab

Designed, hardened and operate a self-hosted Linux server running 6+ containerised services — applying real-world security practices: key-only SSH, firewalling, intrusion prevention, 2FA, network segmentation, TLS, and a 3-2-1 backup strategy.

I designed and built a private Linux server (Proxmox) running six isolated, containerised services, and I secure and maintain it as an ongoing home lab. I hardened it the way you'd protect a real environment: key-only SSH, a default-deny firewall, fail2ban intrusion prevention, two-factor authentication, network segmentation, TLS everywhere, and a Cloudflare WAF with rate-limiting on logins. I also built a 3-2-1 backup and disaster-recovery strategy with automated, scripted backups — and, fittingly, this very website is served from it.

Full case study

The goal

Replace third-party cloud services (media, photos, files) with a private, self-owned, secure home server — and use it as a live environment to build practical cybersecurity and Linux administration skills.

What I built

  • Platform: Proxmox VE (enterprise type-1 hypervisor) on a Dell OptiPlex — Intel i5-9500, 16 GB RAM, multi-disk storage.
  • Architecture: 6 unprivileged LXC containers, each service isolated to contain the blast radius of any compromise, plus the hardened host.
  • Services: Plex (hardware-transcoded media streaming), AdGuard Home (network-wide DNS filtering of ads/trackers/malware), Immich (private, LAN-only photo backup), Kavita (e-book library), Nginx Proxy Manager (reverse proxy with TLS termination), and a media-automation stack.

Security hardening

  • Access control: key-only SSH (passwords disabled), hardened sshd, TOTP 2FA on the management interface, least-privilege read-only API tokens.
  • Network: default-DROP host firewall, services restricted to the LAN subnet, segmentation via isolated containers, minimal audited port-forwarding, unnecessary services disabled.
  • Intrusion prevention: fail2ban auto-bans brute-force attempts — and I review the logs.
  • Perimeter: Cloudflare proxy hides the home IP, with WAF managed rules, Bot Fight Mode and rate-limiting on login endpoints; Let's Encrypt TLS with automated renewal; custom domain with dynamic DNS.
  • Data & endpoint: full-disk encryption on the admin workstation, secure credential handling, and a data-segregation policy for the most sensitive documents.
  • Operations: regular patching cadence and security audits of the host and every container.

Data protection & recovery

  • 3-2-1 backups: automated weekly full-system backups to a dedicated SSD, photo libraries mirrored to a separate drive, an off-site cloud copy, and deletion-protection archives.
  • Reliability: SMART disk-health monitoring, UUID-based mounts, storage quotas, and cron-driven scripted automation.

Retro Snake Arena — Multiplayer Web Game + Security Review

Built and security-reviewed a real-time multiplayer browser game (Node.js + WebSockets), implementing secure authentication, anti-cheat and input validation — and produced a written vulnerability assessment with a remediation plan.

A real-time multiplayer browser game with server-authoritative logic so clients can't cheat. Built from fundamentals — Node.js, Express, WebSockets, vanilla JavaScript and HTML5 Canvas, no framework — with user accounts, multiplayer rooms, moderated in-game chat and leaderboards. An AI-assisted personal project: I directed the work, did the security review, and managed the project end to end.

Full case study

Security engineering

  • Authentication done right: passwords hashed with scrypt + per-user salt and timing-safe comparison; session tokens stored only as SHA-256 hashes; HttpOnly / SameSite cookies.
  • Web-vuln defences: parameterised SQL (injection prevention), input sanitisation and XSS output-escaping, allowlists for user input, and rate-limiting on authentication and actions.
  • Security review: I reviewed the codebase and found real issues — an unauthenticated score-submission endpoint, a missing WebSocket payload cap (DoS risk), and a rate-limit that would be evaded behind a CDN — then wrote a prioritised remediation plan.

Extras

  • Version-controlled on GitHub with incremental, documented commits.
